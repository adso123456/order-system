# Java 后端开发 · 全景学习地图与实战路线

> **目标**：零基础起步，用「电商订单系统」这个项目，完整走一遍 Java 后端开发该会的技术栈。
> **节奏**：分两遍。第一遍**扎实够用**（单体应用，主流技术栈跑通）；第二遍在第一遍基础上**重构**，引入消息队列、分布式等进阶内容。
> **怎么用**：先读 Part A 建立全景认知（知道每个东西是干嘛的），再按 Part B 的路线一步步动手。遇到陌生词回 Part C 速查。

# Part A · 技术栈全景图

后端开发本质上就一件事：**客户端（浏览器/App）发来请求，服务器处理后返回数据**。下面按"一个请求从进来到出去会经过哪些层"来组织，每块说清三件事：是什么 / 为什么需要 / 学到什么程度。

## 0. 概念地基（零基础必读）

在碰任何框架之前，先建立这些基本认知，否则后面全是黑盒：

- **客户端 / 服务端**：你写的后端程序跑在服务器上，等着别人来请求。浏览器、手机 App、Postman 都是客户端。
- **HTTP**：客户端和服务端之间说话的"协议"。核心是请求方法（GET 查、POST 增、PUT 改、DELETE 删）、状态码（200 成功、404 没找到、500 服务器错）、请求体/响应体。
- **REST API**：一种设计接口的风格。比如 `GET /products/1` 表示"获取 id 为 1 的商品"。后端开发大部分时间就是在设计和实现这些接口。
- **JSON**：客户端和服务端传数据的格式，长得像 `{"id":1,"name":"手机"}`。后端把对象转成 JSON 返回，框架会自动帮你做。
- **数据库**：数据不能只存在内存里（程序一关就没了），要持久化到数据库。
- **前后端分工**：前端负责页面和交互，后端负责数据和业务逻辑，两边通过 REST API 对接。本项目只做后端，前端用 Postman / Swagger 模拟。

> **学到程度**：能说清一个请求从浏览器到数据库再回来的完整链路即可，不需要深究 HTTP 底层。

## 1. 语言与工具链

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **Java 17** | 编程语言（LTS 长期支持版） | 后端主流语言之一，生态成熟 | 语法、面向对象、集合、异常、泛型、Lambda；不用啃并发底层 |
| **JDK** | Java 开发工具包（编译+运行环境） | 没它跑不了 Java | 会装、会配 `JAVA_HOME` 即可 |
| **Maven** | 构建+依赖管理工具 | 帮你下载第三方库、打包项目 | 看懂 `pom.xml`，会加依赖、会打包 |
| **IDE (IntelliJ IDEA)** | 写代码的编辑器 | 提示、调试、运行一条龙 | 会建项目、会用断点调试（**调试能力非常重要**） |

## 2. 框架核心：Spring Boot

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **Spring** | Java 后端最主流的框架 | 几乎所有 Java 后端岗位都要 | 理解两个核心思想见下 |
| **Spring Boot** | 让 Spring 开箱即用的脚手架 | 不用手动配一堆 XML，几行就能跑起来 | 会用 Spring Initializr 建项目、会配 `application.yml` |
| **IoC / DI（控制反转/依赖注入）** | 对象不由你 new，交给 Spring 管理和"注入" | Spring 的灵魂，理解了它后面全通 | 必须吃透 `@Component`/`@Service`/`@Autowired` |
| **分层架构** | Controller→Service→Repository 三层 | 职责分离，代码清晰好维护 | 必须养成习惯，这是后端代码的基本骨架 |

> 分层的含义：**Controller** 接请求、出响应（不写业务）；**Service** 写业务逻辑、管事务；**Repository** 只和数据库打交道。

## 3. Web 层

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **Spring MVC** | 处理 HTTP 请求的模块 | 写 REST 接口靠它 | `@RestController`/`@GetMapping`/`@PostMapping`/`@RequestBody`/`@PathVariable` |
| **DTO** | 专门用于接口传输的对象 | 不要把数据库实体直接暴露给前端 | 理解 Entity 和 DTO 为什么要分开 |
| **参数校验** | 自动检查请求参数合法性 | 不能信任前端传来的任何数据 | `@Valid` + `@NotNull`/`@Min` 等注解 |
| **统一返回 & 全局异常** | 所有接口返回同一种结构、异常集中处理 | 前端好对接，代码不重复 | `@RestControllerAdvice` 兜住所有异常 |

## 4. 数据层（最重要的一块）

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **MySQL** | 关系型数据库 | 存数据，企业最常用 | 建库建表、增删改查、索引基础 |
| **SQL** | 操作数据库的语言 | 数据库的"普通话" | 会写 SELECT/INSERT/UPDATE/DELETE、JOIN、WHERE |
| **JPA / Hibernate** | 对象关系映射（ORM） | 让你用 Java 对象操作数据库，少写 SQL | 实体映射、关联关系、`Repository` 接口 |
| **事务（@Transactional）** | 一组操作要么全成功要么全回滚 | 保证数据一致性（如下单扣库存） | 必须理解，本项目核心之一 |
| **连接池（HikariCP）** | 复用数据库连接 | 频繁建连接很慢 | 知道它存在、Spring Boot 默认带，会配几个参数 |

> 进阶可了解 **MyBatis**（另一种持久层框架，SQL 写得更直接），第二遍可以拿一个模块换成它体会差异。

## 5. 缓存

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **Redis** | 内存数据库 | 快，用来缓存热点数据、做延迟队列 | String/Hash/ZSet 等基本类型、TTL 过期、缓存读写模式 |

> 本项目的"订单超时自动取消"就用 Redis 的 ZSet 实现，是你额外感兴趣的点，放在第一遍后期。

## 6. 安全

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **Spring Security** | 认证授权框架 | 控制谁能访问哪些接口 | 过滤器链、`SecurityContext`、放行/拦截规则 |
| **JWT** | 一种登录令牌 | 无状态登录，前后端分离主流方案 | 登录签发 token、请求带 token 校验 |
| **BCrypt** | 密码加密算法 | 密码绝不能明文存 | 注册时加密、登录时比对 |

## 7. 质量与协作

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **JUnit 5** | 单元测试框架 | 验证代码正确、改代码不怕坏 | 会给 Service 写测试 |
| **Mockito** | 模拟依赖的测试工具 | 测 Service 时不真连数据库 | 会 mock Repository |
| **springdoc-openapi (Swagger)** | 自动生成接口文档 | 接口可视化、可在线调试 | 引入依赖即可，会用 Swagger UI 调接口 |
| **日志（SLF4J + Logback）** | 记录程序运行信息 | 排查问题靠它，不靠 `System.out` | 会打不同级别日志、会看日志定位问题 |
| **Git** | 代码版本管理 | 必备基本功 | commit/push/分支基本操作 |

## 8. 部署与运维（第一遍末尾）

| 项 | 是什么 | 为什么 | 学到程度 |
|----|--------|--------|----------|
| **Docker** | 容器，把应用和环境打包 | "在我电脑能跑"问题的终结者 | 会写 Dockerfile、会 build/run |
| **Docker Compose** | 一键启动多个容器 | 同时拉起 app + MySQL + Redis | 会写 `docker-compose.yml` |

## 9. 第二遍才碰的进阶内容

第一遍是**单体应用**，下面这些第二遍重构时再上，否则零基础会被淹没：

| 项 | 是什么 | 解决什么问题 |
|----|--------|-------------|
| **消息队列（RabbitMQ / Kafka）** | 异步通信中间件 | 解耦、削峰、异步处理（如下单后异步发通知、用 MQ 做延迟取消） |
| **Redisson** | Redis 的 Java 增强客户端 | 分布式锁、生产级延迟队列 |
| **分布式锁** | 跨实例的互斥 | 多实例下防止重复处理 |
| **可观测性（日志/指标/链路追踪）** | 监控系统健康 | 线上排查问题（Prometheus、ELK、SkyWalking 等） |
| **CI/CD** | 自动构建部署 | 提交代码自动测试、打包、上线（GitHub Actions） |
| **服务拆分思想** | 了解微服务概念 | 理解为什么以及何时要拆，不一定真拆 |

# Part B · 实战推进路线

> 零基础友好，从装环境讲起。**严格按顺序，前一步没通就别进下一步。** 时间是参考，按自己节奏来。

## 第一遍 · 扎实够用（单体应用）

### 阶段 0 · 环境与概念（约 1 周）
1. 装 **JDK 17**，配好 `JAVA_HOME`，命令行能跑 `java -version`。
2. 装 **IntelliJ IDEA**（社区版免费）。
3. 装 **MySQL** 和 **Redis**（建议用 Docker 装，省事，也顺便认识 Docker）。
4. 装 **Postman**（调接口用）。
5. 补 Java 基础：变量、类、对象、集合（List/Map）、异常处理。零基础的话这部分单独花时间过一遍。
6. 读懂 Part A 第 0 节，能讲清"一个请求的完整链路"。

✅ 出阶段标准：能在 IDEA 里跑一个打印 Hello World 的 Java 程序，能连上自己的 MySQL 和 Redis。

### 阶段 1 · 第一个 Spring Boot 接口（约 1 周）
1. 用 [Spring Initializr](https://start.spring.io) 建项目，勾 Web + JPA + MySQL Driver。
2. 写一个 `@RestController`，实现 `GET /hello` 返回一句话，用 Postman 调通。
3. 理解项目分层，建好 controller/service/repository/entity/dto 目录。
4. 理解 IoC/DI：写个 Service，在 Controller 里注入它。

✅ 出阶段标准：能讲清这个请求是怎么从 Controller 流到 Service 再返回的。

### 阶段 2 · 商品 CRUD（约 1.5 周）
1. 设计 `product` 表，写 Entity。
2. 用 JPA Repository 实现商品的增删改查接口。
3. 加 DTO，分离 Entity 和接口对象。
4. 加 `@Valid` 参数校验。
5. 加全局异常处理 + 统一返回结构。

✅ 出阶段标准：商品五个接口全部 Postman 调通，传非法参数会被正确拦截。

### 阶段 3 · 下单 + 库存一致性（约 1.5 周）
1. 设计 `orders` + `order_item` 表。
2. 实现下单接口：建订单（待支付）+ 扣库存 + 写明细，**全程一个事务**。
3. 用**乐观锁**（version 字段）防止超卖。
4. 实现模拟支付接口：把订单改成已支付。

✅ 出阶段标准：理解 `@Transactional`；高并发下单不会把库存扣成负数。

### 阶段 4 · 取消订单（幂等）（约 1 周）
1. 实现用户主动取消：**条件更新**（`WHERE status='PENDING'`）+ 恢复库存，同一事务。
2. 重复调用取消接口，结果一致（幂等）。

✅ 出阶段标准：理解为什么不能"先查后改"，理解幂等的含义。

### 阶段 5 · 超时自动取消（你感兴趣的点）（约 1 周）
1. 下单时把订单加入 Redis ZSet 延迟队列（score = 到期时间）。
2. 写定时任务轮询到期订单，复用阶段 4 的取消逻辑。
3. 验证：支付后到期的订单会被状态校验自动跳过。

✅ 出阶段标准：能讲清 ZSet 延迟队列的原理和它的局限。

### 阶段 6 · 用户与认证（约 1.5 周）
1. 设计 `users` 表，注册时密码 BCrypt 加密。
2. 登录成功签发 JWT。
3. 引入 Spring Security，区分公开接口和需登录接口。
4. 订单只能操作自己的（鉴权）。

✅ 出阶段标准：理解过滤器链、无状态登录流程。

### 阶段 7 · 缓存 + 文档 + 测试 + 容器化（约 2 周）
1. 用 Redis 缓存商品详情，体会缓存读写和过期。
2. 引入 springdoc，用 Swagger UI 调接口。
3. 给下单、取消等关键 Service 写单元测试（JUnit + Mockito）。
4. 全程接入日志，养成看日志排查的习惯。
5. 写 Dockerfile + docker-compose.yml，一键拉起 app + MySQL + Redis。

✅ **第一遍完成标准**：一个功能完整、能容器化部署、有文档有测试的单体后端项目。到这里你已经是合格的初级后端了。

## 第二遍 · 重构 + 分布式进阶

> 在第一遍的代码基础上改造。目标是体会"为什么要引入这些更复杂的东西"。

### 进阶 1 · 用消息队列重做异步逻辑
- 引入 **RabbitMQ**（或 Kafka）。
- 把"下单成功后发通知"改成发消息异步处理，体会**解耦**。
- 把超时取消从"ZSet 轮询"改成 **RabbitMQ 延迟队列 / 死信队列**，体会专业延迟方案。

### 进阶 2 · 分布式安全
- 用 **Redisson** 替换手写 ZSet，体会生产级延迟队列。
- 给关键操作加**分布式锁**，模拟多实例部署下的并发问题并解决。

### 进阶 3 · 可观测性
- 接入结构化日志，了解 **ELK** 思路。
- 加健康检查（Spring Boot Actuator）和基础指标（Prometheus）。

### 进阶 4 · 自动化
- 用 **GitHub Actions** 配 CI：提交代码自动跑测试、打镜像。

### 进阶 5 · 选学
- 拿一个模块从 JPA 换成 **MyBatis**，对比体会。
- 读一点微服务概念，理解单体 vs 微服务的取舍（不一定真拆）。

# Part C · 零基础概念速查

遇到陌生词回这里查：

- **API**：接口，程序之间约定好的调用方式。后端就是提供 API 给前端调。
- **框架**：别人写好的一套通用代码骨架，你往里填业务。Spring 就是框架。
- **依赖 / 库**：别人写好的可复用代码包，用 Maven 引入。
- **注解（@开头的东西）**：贴在代码上的"标签"，告诉框架这段代码的角色或行为。
- **Bean**：被 Spring 容器管理的对象。
- **实体 Entity**：和数据库表一一对应的 Java 类。
- **ORM**：把数据库的行和 Java 对象互相转换的技术（JPA 干的事）。
- **事务**：一组数据库操作的整体，要么全成功要么全失败。
- **幂等**：同一个操作执行一次和执行多次，结果一样。
- **乐观锁 / 悲观锁**：两种防并发冲突的思路，乐观锁假设冲突少（用版本号），悲观锁假设冲突多（直接锁住）。
- **缓存**：把常用数据放在更快的地方（Redis/内存），减少查数据库。
- **中间件**：独立于你应用的通用服务，如数据库、Redis、消息队列。
